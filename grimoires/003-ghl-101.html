The Speedrun: Zero → Live Automation in 12 Steps
A blank agency account to a tested, working lead-capture system. Every button. ~90 minutes.
  1. Log in at app.gohighlevel.com (agency owners). Clients use their sub-account link or the LeadConnector app. Enable 2FA immediately on all admin logins.
  2. Create a sub-account: Agency View → Accounts → + Add Account → choose template or Start From Scratch → fill Business Name, Contact Info, Time Zone (wrong timezone = every scheduled send fires at the wrong hour), Currency → Save. Never build in the Agency account directly.
  3. Connect your domain: Settings → Domains → Add New Domain. GHL can also register a domain natively (2025) — no need to leave the platform.
  4. Set up email: Settings → Email Services → LC Email for quick start, or Mailgun for deliverability control. Verify SPF, DKIM, and DMARC before any send.
  5. Set up phone: Settings → Phone Numbers → LC Phone (fastest, recommended) or Twilio if you need direct number ownership. Enable STOP keyword handling here too.
  6. Build a calendar: Settings → Calendars → + New Calendar → pick type (User / Round-Robin / Unassigned — see M6) → set availability and buffer times.
  7. Create a pipeline: Opportunities → Pipelines → + Create Pipeline → stages that describe events, not hopes: New Lead → Contacted → Booked → Showed → Won/Lost.
  8. Build a form: Sites → Forms → + Add Form → 2 visible fields max above the fold + hidden UTM fields (M7) + consent checkbox (not pre-ticked).
  9. Build the workflow: Automation → Workflows → + Create Workflow → Trigger: Form Submitted → Actions: Add Tag src:website → Send SMS (instant) → Send Email → Create Opportunity → Internal Notification. Name it properly: WF Lead Capture v1.
  10. Set reentry: in workflow settings, choose Once every X days for nurture flows so a re-submitting contact doesn't get double-enrolled.
  11. Test with a dummy contact: submit the form yourself → watch it execute in Workflow → History tab → confirm tag, SMS, email, opportunity, and notification all fired. Use 1–3 min delays during testing only.
  12. Go live: extend delays to production values → Publish → submit one more real test → check History again. You now have a working system. Everything after this page is depth.
Checkpoint. If step 11 worked end-to-end, you've already done what most $500 GHL courses teach. The remaining modules cover every button behind each of these steps, plus the AI layer.

M2
Agency, Sub-Accounts & Snapshots
The two-layer architecture every other decision sits on

GHL has two layers: the Agency account (your master login — billing, white-label, user permissions) and Sub-accounts (one per client or brand — all CRM data, workflows, funnels, calendars). Sub-accounts are fully isolated from each other by design.

⚡ Power Tip

Plan sub-account structure before building anything. Merging contacts across sub-accounts is painful. Multiple brands sharing one contact list = one sub-account. Genuinely separate businesses = separate sub-accounts. And never build in the Agency layer directly — always work inside a sub-account, even for tests.

Create a Sub-Account

  1. Agency View → Accounts → + Add Account
  2. Choose a template or Start From Scratch
  3. Fill in: Business Name, Contact Info, Time Zone, Currency
  4. Click Save

Snapshots — Productize Your Builds

Snapshots export a sub-account's entire configuration and import it into a new one. This is how one good build becomes every client's starting point.

  1. Agency View → Snapshots → Import Snapshot
  2. Paste the shared snapshot URL
  3. Assign the imported snapshot to a sub-account
Snapshot includes:                    Snapshot excludes:
  ✓ Workflows (trigger logic intact)     ✗ Contacts and CRM data (by design)
  ✓ Funnels and websites                 ✗ Active campaign enrollments
  ✓ Email / SMS templates                ✗ API keys, third-party credentials
  ✓ Custom fields and values             ✗ Reviews / reputation data
  ✓ Pipelines and calendars
Build snapshot-first. Maintain one master snapshot, clone it for every new client, and run a backup snapshot monthly. Name assets descriptively — Landlord D0 Sequence, never Workflow 1 — because snapshot imports preserve names, and bad names multiply.

M4
Contacts, Fields & Tags
The contact record is the source of truth. Treat it like one.

Every automation, workflow, and pipeline stage anchors to a Contact record. The contact data model is the most important structural decision in any GHL build.

The Contact Profile (Updated 2025)

Contacts → select a contact — each profile contains: General Info (name, phone, email, tags, assigned user), Opportunities (pipeline status + deal value), Activity Timeline (forms, page visits, messages, calls), Notes, Tasks, Appointments, and AI Summary 2026 — one-click summary of all conversations: Contact Profile → Activity tab → Generate AI Summary.

⚡ Power Tip — The Single-Owner Rule

Every field gets exactly one writer — one workflow, one form, or one integration. If two workflows both write to lead_score, you get race conditions that are nearly impossible to debug. And group fields before creating them (e.g., "Lead Qualification," "Automation State") — an ungrouped list at 50+ fields becomes unusable in weeks.

Tag Taxonomy — The Segmentation Engine

Tags are the most flexible tool in GHL and the most abused. Without a naming convention you'll have 200 uninterpretable tags within six months.

// Prefix by function. Document every tag in your SOP hub.
SOURCE: src:facebook | src:google | src:referral | src:website
STATUS: status:hot-lead | status:warm | status:cold | status:suppressed
ACTION: action:booked | action:replied | action:clicked | action:converted
SEQ:    seq:active | seq:completed | seq:paused
AUDIT:  review:needed | duplicate:flagged | archive:unqualified

Rules: tags are for logic and filtering, never decoration. Clean unused tags monthly. Smart Lists + tags = custom CRM views per team role.

Duplicates & GDPR Deletion

GHL deduplicates by email and phone by default. A repeat submission merges to the existing record — or creates a duplicate if the phone doesn't match. Always gate Day-0 enrollment on a seq:active tag check (M9, Rule 3).

  1. Stop all active workflows for the contact
  2. Remove them from all active broadcasts
  3. Search contact → open profile → Trash icon → confirm
Deleted data is unrecoverable unless backed up. Soft-archive first with a tag like archive:unqualified before any permanent deletion.

M5
Pipelines & Opportunities
A pipeline is a model of your sales process — not a wishlist

An Opportunity is a deal entry in a pipeline — tracking value, stage, assigned user, and status (Open / Won / Lost). Contacts sit in the CRM; Opportunities are what actually move. One contact can hold multiple opportunities in multiple pipelines simultaneously.

⚡ Power Tip

Stage names describe what happened, not what you hope happens. "Proposal Sent" is a fact. "Interested" is a feeling. Fact-based stages make pipeline reports mean something. And assign a monetary value to every opportunity — that's what unlocks Forecast View and real ROI tracking.

Pipeline Views

ViewUse
KanbanDrag-and-drop board. Best for sales teams tracking live movement.
Forecast ViewExpected revenue by stage. Spot bottlenecks where value pools.
List View 2025SmartList-style with customizable columns and fast filtering.

Stage Triggers (pipeline-driven automation)

  1. Create workflow with trigger: Opportunity Status Changed
  2. Filter by pipeline and specific stage
  3. Add actions: email, SMS, task, internal notification

Lead Scoring (custom-field pattern)

Custom field: Lead Score (Number)
  Clicked email   → +10
  Booked call     → +25
  No show         → −10
Routing: If/Else on Lead Score → high scores to senior reps, low to nurture

Won / Lost Tracking

  1. Open the opportunity card
  2. Change Status dropdown to Won or Lost
  3. Add a note with the reason — required for meaningful reporting
Tag your losses by reason — Lost:Price, Lost:No-Response — so future reactivation campaigns can target each segment with the right message.

Forms & Surveys
The entry point for every lead. Build it like one.

Forms capture lead data and fire automations. Built inside Sites → Forms — drag-and-drop builder with conditional field logic. Embed on external sites via code snippet. Submissions live in Sites → Forms → Submissions.

The Form Standard (use on every form)

MOBILE-FIRST ORDERING:
  Above fold: 2 fields max (name + phone, or email + city)
  Below fold: qualification fields
  // Micro-forms reduce abandonment 30–50% on cold traffic
HIDDEN FIELDS (mandatory for attribution):
  utm_source / utm_medium / utm_campaign → map to contact custom fields
  // Auto-populate from URL params. Without these, paid and organic look identical.

  Consent checkbox — NOT pre-ticked (GDPR/TCPA)
  Privacy policy link — visible, not buried
BACKUP:
  Form Settings → notification email to your own inbox
  // Fires independently of workflows — your lead alert survives a GHL outage

M9
Workflows: The Engine
If you master one module in this manual, master this one

Workflows are GHL's automation engine: a trigger (what starts it), conditions (if/else logic), actions (what happens), and wait/delay steps. The interface: left column = Triggers · center canvas = Steps & Logic · top bar = Publish, Save, Test · right panel = Action settings.

The Five Architecture Rules

RULE 1: One workflow, one job.
  No 40-step mega-workflows covering 3 scenarios. Split by function:
  D0-Instant-Response · Day-3-Follow-Up · Booking-Confirmed.
  Single-job workflows are testable, debuggable, replaceable.
RULE 2: Name everything descriptively. Use folders.
  Bad:  Workflow 1 | New Automation | Copy of Copy of Test
  Good: [Booking] WF FB Lead → Booking v2 | [Nurture] WF Cold Reactivation v3
  Folders by function: Booking, Nurture, Internal, AI.
RULE 3: Always add a loop guard.
  Step 1 of any enrollment workflow: check for seq:active tag.
  Tagged → stop or alert. Prevents double-enrollment on re-submission.
RULE 4: Test with a dummy contact before going live.
  Create test contact → fire the trigger manually → watch History tab live.
RULE 5: Wait steps for time-sensitive sends.
  Wait → Send beats scheduling inside the action — it gives you a cancel window.

Reentry Settings

SettingBehavior
Multiple entries (default)Contact enters every time the trigger fires
Only once per contactOne entry, ever
Once every X days/hoursTime-limited reentry — ideal for drip campaigns
Reentry is the silent killer both ways: default multiple-entry double-enrolls repeat submitters; "only once" makes re-engagement loops fire exactly once and never again. Choose deliberately, per workflow.

Troubleshooting — The History Tab Routine

  1. Automation → Workflows → Open Workflow → History tab
  2. Search the contact's email or phone
  3. Check: Did it trigger? What was the last completed step? Were steps skipped?
  4. Logs tab: Twilio bounces, invalid emails, API errors
  5. Check Status column: Paused / Completed / Failed — paused on a Wait step usually means the required condition was never met (the #1 cause of stalled workflows)
⚡ Power Tip

Two debugging force-multipliers: (1) add Internal Notification steps between logic blocks during QA builds — you literally watch the workflow walk; (2) enable workflow error notification emails and check them daily until the system proves stable. Workflows fail silently by default. Don't let them.

"Tag Added" is the most versatile trigger in GHL. Any workflow, Zap, n8n node, or API call can apply a tag — and a separate workflow reacts to it. Systems talk through tags instead of nested logic. This is decoupled architecture inside a no-code tool.

Payments & SaaS Mode
Gate on payment.cleared. Everything else follows.

Payment Integrations

Connect Stripe: Settings → Integrations → Connect Stripe. PayPal and Venmo support one-click OAuth setup Dec 2025. GHL's open payments infrastructure also supports NMI, Authorize.net, Square, Adyen, Razorpay, and custom providers Verified Jun 2026. Use in funnel order forms, invoices, and subscription plans.

PAYMENT EVENT TRIGGERS:
  payment.cleared      → ✓ Move to Closed-Won, fire fulfillment, send receipt
  payment.submitted    → ⚠ NEVER use for fulfillment (funds not confirmed)
  payment.failed       → SMS alert + pause delivery + collections sequence
  subscription.created   → start retention sequence
  subscription.cancelled → cancellation flow, offboarding
  Abandoned Cart        → recovery sequence
The most expensive GHL mistake: fulfilling on payment.submitted (card entered) instead of payment.cleared (funds confirmed). Cards decline. Payments fail. Gate everything on cleared — no exceptions. And build the Payment Failed workflow, or failed subscriptions disappear into silence.

↺ The most common failure in real builds

Workflows edited live without testing. One change to a merge tag, trigger filter, or wait duration silently breaks every future enrollment. Make the change in a test workflow → QA it → copy the fix to production. Ten extra minutes; hours of cleanup saved, every time.
